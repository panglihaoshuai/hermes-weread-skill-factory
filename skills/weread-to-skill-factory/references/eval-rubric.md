# 评估评分标准（Eval Rubric）

每个 book-skill 必须通过三项评估才能发布。

## 1. 触发测试（trigger-tests.md）

### 评估维度

| 维度 | 标准 | 通过条件 |
|------|------|---------|
| 准确率 | 应该触发的输入确实触发了 | ≥ 80% |
| 误触发率 | 不该触发的输入没有触发 | ≤ 10% |
| 边界清晰度 | 触发条件是否明确 | 无歧义 |

### 测试用例设计

**应该触发（8-10个）：**
- 直接引用书名/作者
- 使用书中概念
- 问题领域与书匹配
- 用户明确要求"用XX框架分析"

**不应该触发（8-10个）：**
- 与书无关的问题
- 类似但不同的领域
- 简单问题不需要框架
- 用户明确要其他风格

## 2. 应用测试（application-tests.md）

### 评估维度

| 维度 | 标准 | 通过条件 |
|------|------|---------|
| 框架化 | 回答使用了核心模型 | 使用 ≥ 2 个模型 |
| 具体性 | 给出了 if-then 规则，不是泛泛建议 | 有具体行动步骤 |
| 诊断性 | 先问诊断问题再给建议 | 有 ≥ 1 个诊断问题 |
| 个性化 | 结合了用户的实际情况 | 提及用户项目/场景 |

### 测试场景设计

每个 book-skill 准备 3 个真实场景：
1. **直接应用**：书中有明确答案的问题
2. **框架迁移**：书中的框架应用于新领域
3. **边界场景**：接近框架边界的灰色地带

### 评分标准

每个场景 1-5 分：
- 1分：泛泛套话，无框架
- 2分：提到了书名但无实质
- 3分：使用了1个模型但不完整
- 4分：使用了2+模型，有诊断问题
- 5分：框架化回答，有行动协议，有诚实边界

通过条件：平均分 ≥ 3.5

## 3. 对抗测试（adversarial-tests.md）

### 评估维度

| 维度 | 测试方法 | 通过条件 |
|------|---------|---------|
| 抗套话 | 问模糊问题 | 不回答"要全面思考"之类 |
| 抗过度自信 | 问超出框架的问题 | 诚实说"这个框架不擅长" |
| 抗复读 | 问书的内容 | 不大段引用原文 |
| 抗泛化 | 问具体场景 | 给具体 if-then，不是"因人而异" |

### 测试用例

1. **模糊问题**："我该怎么做决策？"
   - 失败："要全面思考，权衡利弊"
   - 成功："先反转问题：如果你不做这个决策，会发生什么？"

2. **超出框架**："用XX框架分析量子力学"
   - 失败：强行分析
   - 成功："这个框架不擅长处理XX，建议用YY"

3. **复读测试**："这本书讲了什么？"
   - 失败：大段引用原文
   - 成功："核心是3个认知模型：...（框架化总结）"

4. **泛化测试**："具体怎么做？"
   - 失败："因人而异"
   - 成功："步骤1→2→3，如果遇到X则调整为Y"

### 通过条件

- 抗套话：≥ 80% 不回答泛泛建议
- 抗过度自信：≥ 90% 诚实承认局限
- 抗复读：≥ 80% 不大段引用原文
- 抗泛化：≥ 70% 给出具体步骤

## 综合评估

| 评估 | 权重 | 通过条件 |
|------|------|---------|
| 触发测试 | 20% | 准确率 ≥ 80%，误触发 ≤ 10% |
| 应用测试 | 50% | 平均分 ≥ 3.5/5 |
| 对抗测试 | 30% | 4项全部通过 |

**总分 ≥ 70%** → 发布
**总分 50-69%** → 标注为 needs-improvement
**总分 < 50%** → 不发布，返回修改

## Pitfalls

### Static Analysis ≠ Real Execution

**问题**：eval 的 Observed 字段只是对 SKILL.md 规则的静态分析，而非实际调用 skill 后观察到的行为。

**反模式**：
```
Observed：根据 SKILL.md 诊断问题 1 → 是。skill 应识别为调查优先场景。
```

**正确做法**：
```
Observed：实际调用 skill，输入"我要开始一个新项目"，
skill 输出包含"先收集信息"和"brainstorming 讨论清楚"。
```

**规则**：
- 如果 Observed 只是"根据 SKILL.md 应该..."，标为 WEAK PASS
- 优先使用 Real Execution（delegate_task / terminal / agent 执行）
- Static Analysis 只能证明"规则写对了"，不能证明"skill 按规则运行"

### Smoke Test Must Have Independent Evidence

**问题**：smoke test 结果只存在于 final report 中，没有独立证据文件。

**规则**：
- smoke test 必须生成独立日志文件（如 `smoke-test-log.md`）
- final report 引用日志文件，不自己证明自己
- 没有独立日志的 smoke test 标为 SMOKE_TEST_EVIDENCE_WEAK

### Eval Template ≠ Eval Result

**问题**：eval 文件只有测试模板（Test Name/Input/Expected），没有 Observed/Result。

**规则**：
- 只有模板没有结果的 eval 文件不得算 PASS
- 必须有 Observed 字段，且 Observed ≠ Expected 的简单复述
- 如果 Observed 照抄 Expected，标为可疑
